ReadCast Architecture

Personal read-later + listen app — high-level stack overview

Overview

ReadCast saves article URLs, extracts readable content, stores it locally, and plays it back as a podcast-style queue. The mobile app (iOS/Android via Expo) talks to an optional Hono backend for article extraction and TTS. In development, DEV_STUB_MODE bypasses the backend entirely.

  User->>UI: paste URL
  UI->>Ingest: saveArticle(url)
  alt DEV_STUB_MODE
    Ingest->>Store: fixture article (ready)
  else production
    Ingest->>Store: pending article
    Ingest->>API: POST /articles/ingest
    API-->>Ingest: title, html, text
    Ingest->>Store: ready article
  end

  User->>UI: tap Listen
  UI->>Audio: generateAudio(article)
  alt DEV_STUB_MODE
    Audio->>Store: sample segments
  else production
    Audio->>API: POST /articles/:id/audio
    API-->>Audio: audio segments
    Audio->>Store: AudioJob + segment URIs
  end
  Audio->>Player: load track + play

Targets

  • Later: Maestro native e2e

Key Design Choices

  • Seams over branches — shared code depends on interfaces (ArticleStore, PlaybackEngine); platform files swap implementations.
  • Stub-first dev — DEV_STUB_MODE runs the full ingest → read → listen loop without backend or API keys.
  • Chunked TTS — long articles split on sentence bounds; backend synthesizes per chunk, app stitches segments into a track.
  • Thin backend — two POST routes; extraction and TTS are the only server-side concerns. All library state lives on-device.
